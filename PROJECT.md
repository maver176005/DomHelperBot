# PROJECT.md

## Product

Telegram-бот для соседского маркетплейса внутри многоквартирных домов.

Главная идея: жители одного дома помогают друг другу за деньги или по договоренности.

Первый MVP-сценарий — вынос мусора от двери.

Дополнительные сценарии:
- услуги соседей;
- аренда вещей;
- поиск навыков внутри дома.

---

## Core Principle

Все действия происходят внутри конкретного дома.

Пользователь из одного дома не должен видеть:
- заказы другого дома;
- исполнителей другого дома;
- вещи другого дома;
- услуги другого дома.

Ключевое поле почти во всех сущностях: `houseId`.

---

## Roles

- `client` — житель, который заказывает услугу.
- `provider` — житель, который выполняет услуги.
- `house_admin` — админ конкретного дома.
- `super_admin` — админ всей платформы.

---

## Main MVP Flow: Trash Removal

1. Клиент открывает бота.
2. Выбирает “Вынести мусор”.
3. Указывает детали:
    - количество пакетов;
    - комментарий;
    - способ оплаты;
    - фото мусора у двери.
4. Бот создает заказ.
5. Исполнители из этого же дома получают уведомление.
6. Первый исполнитель нажимает “Взять заказ”.
7. Заказ закрепляется за ним.
8. Исполнитель видит точные данные.
9. После выполнения исполнитель отправляет фото после.
10. Клиент подтверждает выполнение.
11. Заказ считается завершенным.

---

## Important Rules

- Исполнитель должен быть жителем того же дома.
- Заказ нельзя взять двум исполнителям одновременно.
- Фото до обязательно для создания заказа на мусор.
- Фото после обязательно для завершения заказа.
- До взятия заказа нельзя показывать точный номер квартиры.
- После взятия заказа можно показать точные детали.
- Все спорные места решать через разумные MVP-упрощения.

## Marketplace Logic

У пользователя есть два основных действия:

### Мне нужно

Пользователь ищет:
- услугу;
- вещь;
- помощь;
- исполнителя.

### Я предлагаю

- могу вынести мусор;
- могу покрасить;
- могу собрать мебель;
- могу дать дрель в аренду.

## MVP Scope

Нужно сделать минимально рабочий продукт.

В MVP должно быть:

- запуск Telegram-бота;
- регистрация пользователя;
- привязка к дому;
- создание заказа на вынос мусора;
- уведомление исполнителей своего дома;
- взятие заказа исполнителем;
- защита от двойного взятия заказа;
- загрузка фото до;
- загрузка фото после;
- подтверждение заказа клиентом;
- базовая заготовка для услуг;
- базовая заготовка для аренды вещей;
- README с запуском.

## Product Decisions

- Telegram — основной интерфейс.
- Сначала делаем бота, не веб-приложение.
- Первый платный сценарий — вынос мусора.
- Исполнители — только жители этого же дома.
- Деньги на MVP: наличка или перевод.
- Онлайн-оплата будет позже.
- Модерация исполнителей нужна, но может быть простой.
- Главный фильтр безопасности — `houseId`.

---

## Anti Goals

Не превращать MVP в огромную платформу сразу.

Не делать:

- маркетплейс на всю страну с первого дня;
- сложную CRM;
- суперадминку на 100 экранов;
- идеальную платежную систему;
- сложный dispute center;
- тяжелый enterprise backend.

---

## Development Style

Пиши код так, чтобы его можно было развивать.

Предпочтения:

- TypeScript;

- простая модульная структура;

# LESSONS.md

Журнал самообучения проекта.

После каждого исправления ошибки или неудачного решения сюда добавляется короткая запись.
Цель: не просто исправить конкретный баг, а превратить его в правило, которое снижает шанс повторения.

## Формат записи

```md
## YYYY-MM-DD — короткое название

### Ошибка

Что пошло не так. Писать конкретно: симптом, место, сценарий, команда или файл.

### Урок

Почему это случилось и какой вывод нужно запомнить.

### Новое правило

Как действовать дальше в похожих ситуациях. Правило должно быть проверяемым и применимым.
```

## Правила ведения

- Запись добавляется после каждого исправления, если в ходе работы была обнаружена ошибка, регрессия, неверное предположение или слабое место процесса.
- Не записывать общие мысли без практического правила.
- Не заменять этим `STATUS.md`, `DECISIONS.md` или changelog: здесь фиксируются именно уроки из ошибок.
- Формулировать так, чтобы следующий исполнитель мог применить правило без восстановления всего контекста.
- Если ошибка связана с командой или тестом, указывать точную команду.
- Если ошибка связана с кодом, указывать файл или модуль.

## 2026-05-02 — Ввести цикл самообучения

### Ошибка

В проекте не было отдельного места, где фиксируются выводы после исправлений. Из-за этого полезные уроки могли оставаться только в переписке и теряться между сессиями.

### Урок

Исправление без зафиксированного правила помогает только один раз. Для долгого проекта нужен короткий журнал, который превращает повторяющиеся ошибки в рабочие ограничения.

### Новое правило

После каждого исправления ошибки проверять, нужен ли новый пункт в `docs/LESSONS.md`. Если нужен, добавить запись в формате `Ошибка / Урок / Новое правило` в той же рабочей сессии.

## 2026-05-02 — Деплой должен быть частью репозитория

### Ошибка

До настройки CI/CD в репозитории не было `.github/workflows/ci.yml`, `railway.json` и инструкции деплоя. Из-за этого обычный цикл `commit -> push -> Railway` зависел бы от ручной настройки и памяти исполнителя.

### Урок

Для повторяемого деплоя недостаточно работающего `npm start`. Нужно хранить рядом с кодом CI workflow, Railway config-as-code и инструкцию по обязательным переменным окружения.

### Новое правило

Если проект должен деплоиться автоматически, добавлять в репозиторий минимальный deployment contract: CI workflow на `push`, platform config-as-code, список env-переменных и шаги проверки после push.

## 2026-05-02 — Проверять Railway-подход по рабочему референсу

### Ошибка

Первичная настройка деплоя опиралась на Railpack autodetect, хотя в похожем рабочем проекте `care-bot` уже используется Dockerfile и отдельный Docker build в CI. Это могло привести к расхождению между локальной проверкой, GitHub Actions и тем, что реально собирает Railway.

### Урок

Если в соседнем проекте уже есть рабочий деплой-стек, его нужно проверить до финализации новой настройки. Dockerfile делает Railway build path явным и убирает часть неопределенности autodetect.

### Новое правило

Перед настройкой CI/CD для похожего проекта проверять существующий успешный референс. Если референс использует Dockerfile, добавлять Docker build в CI и явно фиксировать Dockerfile deploy в platform config-as-code.

## 2026-05-02 — Не заявлять локальную Docker-проверку без Docker CLI

### Ошибка

После добавления Dockerfile была запущена команда `docker build -t dom-helper-bot:ci .`, но на локальной машине Docker CLI отсутствует: `zsh:1: command not found: docker`.

### Урок

CI может проверить Docker build на GitHub runner, даже если локальная машина не умеет запускать Docker. Но в отчете нельзя писать, что локальный Docker build пройден, если команда не выполнялась.

### Новое правило

Если локальная инфраструктурная команда недоступна, явно фиксировать это в результате и переносить проверку в CI. В документации помечать такую локальную команду как опциональную при наличии соответствующего инструмента.

## 2026-05-02 — После live deploy нужен явный следующий шаг

### Ошибка

После успешного Railway deploy можно ошибочно продолжить улучшать второстепенные части продукта, хотя главный риск уже стал очевиден: файловое `data/db.json` не является надежным хранилищем на Railway.

### Урок

Live deploy меняет приоритеты. После выхода в Railway следующим обязательным шагом становится не новая функция, а постоянное хранение данных.

### Новое правило

После успешного deploy фиксировать handoff-файл `docs/NEXT_SESSION.md` с первым следующим шагом. Если проект все еще использует файловое хранилище на PaaS, первым шагом следующей сессии должна быть постоянная БД или явно выбранное временное хранилище.

## 2026-05-02 — План БД должен сохранять текущий storage API

### Ошибка

Перед подключением БД можно начать переписывать Telegram handlers напрямую под SQL. В текущем коде много вызовов `readDb()` и `withDb()`, поэтому такой подход дал бы большой риск регрессий в уже работающем Railway-боте.

### Урок

Когда продукт уже работает, инфраструктурную замену хранилища нужно делать через совместимый слой. Сначала меняется реализация storage, а не пользовательские сценарии.

### Новое правило

При миграции хранилища сохранять публичный API текущего storage-слоя на первом этапе. Для `DomHelperBot` Postgres должен сначала поддержать `ensureDb/readDb/writeDb/withDb` и текущий shape `{ houses, users, orders, listings }`.

## 2026-05-02 — Полный редизайн не равен итерации

### Ошибка

При просьбе полностью переделать лендинг была сделана переработка поверх той же визуальной ДНК: сохранились близкие шрифты, темная палитра, карточная структура и общий принцип расположения блоков.

### Урок

Если пользователь просит полностью переделать дизайн, нужно менять не отдельные детали, а базовую дизайн-систему: типографику, палитру, сетку, навигацию, композицию и визуальные метафоры.

### Новое правило

Перед полным редизайном явно проверить, что новая версия не наследует прежние шрифты, цвета, основные layout-паттерны и компоненты. Если наследование остается, это не полный редизайн, а итерация.

## 2026-05-02 — Секреты не должны попадать в репозиторий

### Ошибка

При подключении Railway Postgres реальный `DATABASE_URL` был передан в рабочем чате. Такой URL содержит логин, пароль, хост и порт БД, поэтому его нельзя вставлять в код, документацию или commit history.

### Урок

Даже если секрет нужен для настройки deploy, репозиторий должен хранить только имя переменной окружения и пример без реального значения. Реальное значение должно жить в Railway Variables или локальном `.env`, который исключен из git.

### Новое правило

Перед каждым коммитом с env-настройками запускать поиск по характерным частям секретов (`postgresql://`, реальные хосты, токены). Если секрет уже попал в публичное место, его нужно ротировать.

## 2026-05-02 — VPS для Telegram нужно проверять по сети до настройки

### Ошибка

При попытке переноса на REG.RU VPS проект был установлен и `.env` настроен, но запуск упал на `ETIMEDOUT` при запросе к `https://api.telegram.org/.../getMe`. Время ушло на серверную настройку до проверки главного риска: доступности Telegram API с выбранного хостинга.

### Урок

Для Telegram-бота на polling публичный IP не нужен, но исходящий доступ к `api.telegram.org:443` критичен. На некоторых VPS/сетях это может быть нестабильно или заблокировано. Ответ поддержки REG.RU подтвердил, что проблема может быть не в конкретном сервере, а в массовом замедлении Telegram на территории РФ через ТСПУ у операторов связи.

### Новое правило

Перед переносом Telegram-бота на новый VPS первым делом проверять сеть командой `curl -I --connect-timeout 10 https://api.telegram.org`. Для production runtime предпочитать хостинг вне проблемной сети или уже проверенную платформу. Только после успешной проверки ставить Node.js, PM2 и деплоить приложение.

## 2026-05-03 — Карточка заказа не должна открывать саму себя

### Ошибка

В сценарии "Мои заказы" список заказов и открытая карточка использовали одну и ту же inline-клавиатуру. Поэтому после нажатия "Открыть заказ" пользователь видел такую же кнопку еще раз, а статусные действия вроде подтверждения выполнения или отметки заказа выполненным были неочевидны.

### Урок

Для списка и детальной карточки заказа нужны разные affordance: в списке уместна кнопка открытия, а внутри карточки должны быть только действия, которые меняют состояние заказа или помогают завершить сценарий.

### Новое правило

При добавлении inline-кнопок разделять навигационные кнопки и статусные действия. Детальный экран сущности не должен содержать кнопку, которая повторно открывает этот же экран; для заказов на карточке должны быть доступны действия, соответствующие текущему статусу.

## 2026-05-03 — Первый запрос после простоя должен переживать холодный коннект БД

### Ошибка

После простоя Railway/Postgres первый клик в боте мог падать с `ETIMEDOUT` или `ECONNREFUSED` на подключении к Postgres, а повторный клик уже проходил. Пользователь видел "Что-то пошло не так", хотя сценарий был рабочим.

### Урок

В production PaaS transient-сетевые ошибки на первом подключении к БД нужно считать нормальным инфраструктурным риском. Если операция еще не начала менять состояние, storage-слой должен сам повторить подключение и сбросить старый pool.

### Новое правило

Для внешних storage-зависимостей добавлять ограниченный retry только на transient connection errors (`ETIMEDOUT`, `ECONNREFUSED`, `ECONNRESET`, `ENETUNREACH`, `EAI_AGAIN`). Не ретраить бизнес-ошибки и SQL-ошибки.

## 2026-05-03 — Polling-бот нельзя держать в Railway Serverless

### Ошибка

В Railway `Serverless` был включен и у `DomHelperBot`, и у Postgres. После простоя первый клик в Telegram мог отвечать ошибкой, а со второго раза сценарий работал. Для polling-бота sleep-режим конфликтует с моделью постоянного `getUpdates`, а sleep Postgres добавляет холодный старт БД.

### Урок

Telegram polling — это long-running worker, а не HTTP-сервис, который можно надежно разбудить входящим запросом. Для такого runtime экономия через sleep ломает пользовательский сценарий сильнее, чем экономит ресурсы.

### Новое правило

Для Telegram polling production держать Railway `Serverless` выключенным у bot service и его основной БД. `Wait for CI` можно и нужно включать, но sleep/serverless использовать только после перехода на webhook/HTTP-модель или отдельный wakeup design.

## 2026-05-03 — Рейтинг должен быть следствием закрытого заказа

### Ошибка

При проектировании доверия можно начать показывать исполнителей без механизма качества. Тогда при росте пользователей клиент не понимает, кого выбирать, а продукт не отличает надежных помощников от случайных.

### Урок

Оценка должна быть привязана к реальному завершенному заказу, а не к профилю сама по себе. Это снижает риск накрутки и делает рейтинг понятным: звезды появляются только после подтвержденной работы.

### Новое правило

Для trust-функций сначала привязывать сигнал качества к завершенному действию. В `DomHelperBot` рейтинг исполнителя считается только по подтвержденным заказам с оценкой клиента 1-5.

## 2026-05-03 — Верификация не должна быть флагом по умолчанию

### Ошибка

В регистрации поле `isResidentVerified` выставлялось в `true`, хотя механики проверки проживания в доме пока нет. Это могло бы создать ложное ощущение безопасности в будущей логике и документации.

### Урок

Данные должны отражать реальный уровень доверия, а не желаемый продуктовый статус. Если проверки нет, пользователь может быть привязан к дому, но не верифицирован как жилец.

### Новое правило

Флаги доверия (`isResidentVerified`, verified badges, доступ модератора) ставить в `true` только после конкретного проверяемого действия. Для текущего MVP новые пользователи регистрируются с `isResidentVerified: false`.

## 2026-05-03 — Seed-данные должны доезжать до существующего production state

### Ошибка

После смены seed-домов на Обнинск в production Postgres остался старый `app_state`, созданный раньше. Из-за этого в регистрации показывалась кнопка "Моего дома нет", но не было списка домов Обнинска.

### Урок

JSONB state-store не обновляется автоматически при изменении `DEFAULT_DB`. Если seed-данные считаются актуальной частью продукта, storage-слой должен мягко добавлять новые seed-записи в существующее состояние, не удаляя пользовательские данные.

### Новое правило

При изменении seed-справочников добавлять merge/migration для уже существующего production state. Никогда не рассчитывать, что `DEFAULT_DB` применится к Postgres после первого создания `app_state`.

## 2026-05-03 — Нормализация ввода не должна превращать мусор в валидные данные

### Ошибка

Валидация телефона была слишком мягкой: нормализация удаляла все нецифровые символы, а проверка смотрела только на длину оставшихся цифр. Такой подход может принять ввод с буквами или случайным мусором, если внутри нашлось достаточно цифр.

### Урок

Нормализация не равна валидации. Для пользовательских данных сначала нужно отсеять запрещенные символы и неверный формат, а уже потом приводить значение к каноническому виду.

### Новое правило

Для регистрационных данных использовать строгие проверяемые форматы. В `DomHelperBot` телефон MVP хранится только как российский `+7XXXXXXXXXX`; короткие номера, буквы и произвольный текст не нормализуются в валидное значение.

## 2026-05-03 — Список с полной карточкой не должен иметь кнопку открытия

### Ошибка

Экран "Мои заказы" показывал полную карточку заказа со статусом и деталями, но все равно добавлял кнопку "Открыть заказ". Пользователь видел уже открытый заказ и справедливо не понимал, зачем открывать его еще раз.

### Урок

Кнопка открытия нужна только в компактном списке, где сама карточка не содержит всех деталей. Если экран уже показывает полную карточку и статусные действия, повторная навигационная кнопка создает шум.

### Новое правило

Для полноразмерных карточек заказов использовать только действия по состоянию заказа: отменить, отметить выполненным, подтвердить, повторить. `Открыть заказ` оставлять только для будущих компактных списков или внешних уведомлений.

## 2026-05-03 — QR должен масштабировать запуск дома, а не зависеть от владельца проекта

### Ошибка

Первичная идея домового QR выглядела как ручной админский процесс: владелец проекта создает QR для дома и сам идет его печатать. Это не работает для людей, которые увидели рекламу в интернете и хотят запустить бот в своем доме без участия владельца.

### Урок

QR и ссылка дома должны быть частью продукта. Житель-инициатор должен получить готовую домовую ссылку и QR сразу после добавления дома или из кнопки "Пригласить соседей".

### Новое правило

Growth-механики делать self-serve. Если пользователь создал дом, бот должен дать ему инструменты для приглашения соседей: домовую ссылку, QR-код и понятный текст, который можно отправить или распечатать.

## 2026-05-03 — Монетизация не должна продавать доверие

### Ошибка

Идею `PRO` для исполнителей можно ошибочно превратить в покупку рейтинга или доверия. Это сломает главный механизм выбора исполнителя, особенно в доме, где безопасность и репутация важнее рекламы.

### Урок

Платный тариф должен продавать расширение возможностей, а не искажение доверия. Для исполнителя честная ценность — больше охват домов, профиль, статистика и видимость при равных условиях, но не купленные оценки.

### Новое правило

Будущая монетизация `PRO` может давать исполнителю дополнительные дома/радиус и инструменты профиля. Рейтинг, отзывы и отметка проживания должны оставаться следствием реальных заказов и проверок, а не оплаты.

## 2026-05-03 — Предзаполненное значение не всегда означает выбор пользователя

### Ошибка

При редактировании профиля старый `houseId` попадал в registration flow как предзаполненное значение. Логика регистрации воспринимала любой `houseId` как уже выбранный дом и пропускала экран выбора дома. Из-за этого пользователь не мог сменить дом через профиль.

### Урок

В flow нужно различать предзаполненные данные и данные, выбранные на текущем шаге. Старое значение из профиля не должно блокировать возможность изменить это поле.

### Новое правило

Для автоподстановок добавлять явный флаг источника. В регистрации дома `houseAutoSelected` используется только для новой регистрации по домовой ссылке; при редактировании профиля выбор дома всегда показывается заново.

# LESSONS.md

Журнал самообучения проекта.

После каждого исправления ошибки или неудачного решения сюда добавляется короткая запись.
Цель: не просто исправить конкретный баг, а превратить его в правило, которое снижает шанс повторения.

## Формат записи

```md
## YYYY-MM-DD — короткое название

### Ошибка

Что пошло не так. Писать конкретно: симптом, место, сценарий, команда или файл.

### Урок

Почему это случилось и какой вывод нужно запомнить.

### Новое правило

Как действовать дальше в похожих ситуациях. Правило должно быть проверяемым и применимым.
```

## Правила ведения

- Запись добавляется после каждого исправления, если в ходе работы была обнаружена ошибка, регрессия, неверное предположение или слабое место процесса.
- Не записывать общие мысли без практического правила.
- Не заменять этим `STATUS.md`, `DECISIONS.md` или changelog: здесь фиксируются именно уроки из ошибок.
- Формулировать так, чтобы следующий исполнитель мог применить правило без восстановления всего контекста.
- Если ошибка связана с командой или тестом, указывать точную команду.
- Если ошибка связана с кодом, указывать файл или модуль.

## 2026-05-02 — Ввести цикл самообучения

### Ошибка

В проекте не было отдельного места, где фиксируются выводы после исправлений. Из-за этого полезные уроки могли оставаться только в переписке и теряться между сессиями.

### Урок

Исправление без зафиксированного правила помогает только один раз. Для долгого проекта нужен короткий журнал, который превращает повторяющиеся ошибки в рабочие ограничения.

### Новое правило

После каждого исправления ошибки проверять, нужен ли новый пункт в `docs/LESSONS.md`. Если нужен, добавить запись в формате `Ошибка / Урок / Новое правило` в той же рабочей сессии.

## 2026-05-02 — Деплой должен быть частью репозитория

### Ошибка

До настройки CI/CD в репозитории не было `.github/workflows/ci.yml`, `railway.json` и инструкции деплоя. Из-за этого обычный цикл `commit -> push -> Railway` зависел бы от ручной настройки и памяти исполнителя.

### Урок

Для повторяемого деплоя недостаточно работающего `npm start`. Нужно хранить рядом с кодом CI workflow, Railway config-as-code и инструкцию по обязательным переменным окружения.

### Новое правило

Если проект должен деплоиться автоматически, добавлять в репозиторий минимальный deployment contract: CI workflow на `push`, platform config-as-code, список env-переменных и шаги проверки после push.

## 2026-05-02 — Проверять Railway-подход по рабочему референсу

### Ошибка

Первичная настройка деплоя опиралась на Railpack autodetect, хотя в похожем рабочем проекте `care-bot` уже используется Dockerfile и отдельный Docker build в CI. Это могло привести к расхождению между локальной проверкой, GitHub Actions и тем, что реально собирает Railway.

### Урок

Если в соседнем проекте уже есть рабочий деплой-стек, его нужно проверить до финализации новой настройки. Dockerfile делает Railway build path явным и убирает часть неопределенности autodetect.

### Новое правило

Перед настройкой CI/CD для похожего проекта проверять существующий успешный референс. Если референс использует Dockerfile, добавлять Docker build в CI и явно фиксировать Dockerfile deploy в platform config-as-code.

## 2026-05-02 — Не заявлять локальную Docker-проверку без Docker CLI

### Ошибка

После добавления Dockerfile была запущена команда `docker build -t dom-helper-bot:ci .`, но на локальной машине Docker CLI отсутствует: `zsh:1: command not found: docker`.

### Урок

CI может проверить Docker build на GitHub runner, даже если локальная машина не умеет запускать Docker. Но в отчете нельзя писать, что локальный Docker build пройден, если команда не выполнялась.

### Новое правило

Если локальная инфраструктурная команда недоступна, явно фиксировать это в результате и переносить проверку в CI. В документации помечать такую локальную команду как опциональную при наличии соответствующего инструмента.

## 2026-05-02 — После live deploy нужен явный следующий шаг

### Ошибка

После успешного Railway deploy можно ошибочно продолжить улучшать второстепенные части продукта, хотя главный риск уже стал очевиден: файловое `data/db.json` не является надежным хранилищем на Railway.

### Урок

Live deploy меняет приоритеты. После выхода в Railway следующим обязательным шагом становится не новая функция, а постоянное хранение данных.

### Новое правило

После успешного deploy фиксировать handoff-файл `docs/NEXT_SESSION.md` с первым следующим шагом. Если проект все еще использует файловое хранилище на PaaS, первым шагом следующей сессии должна быть постоянная БД или явно выбранное временное хранилище.

## 2026-05-02 — План БД должен сохранять текущий storage API

### Ошибка

Перед подключением БД можно начать переписывать Telegram handlers напрямую под SQL. В текущем коде много вызовов `readDb()` и `withDb()`, поэтому такой подход дал бы большой риск регрессий в уже работающем Railway-боте.

### Урок

Когда продукт уже работает, инфраструктурную замену хранилища нужно делать через совместимый слой. Сначала меняется реализация storage, а не пользовательские сценарии.

### Новое правило

При миграции хранилища сохранять публичный API текущего storage-слоя на первом этапе. Для `DomHelperBot` Postgres должен сначала поддержать `ensureDb/readDb/writeDb/withDb` и текущий shape `{ houses, users, orders, listings }`.

## 2026-05-02 — Полный редизайн не равен итерации

### Ошибка

При просьбе полностью переделать лендинг была сделана переработка поверх той же визуальной ДНК: сохранились близкие шрифты, темная палитра, карточная структура и общий принцип расположения блоков.

### Урок

Если пользователь просит полностью переделать дизайн, нужно менять не отдельные детали, а базовую дизайн-систему: типографику, палитру, сетку, навигацию, композицию и визуальные метафоры.

### Новое правило

Перед полным редизайном явно проверить, что новая версия не наследует прежние шрифты, цвета, основные layout-паттерны и компоненты. Если наследование остается, это не полный редизайн, а итерация.

## 2026-05-02 — Секреты не должны попадать в репозиторий

### Ошибка

При подключении Railway Postgres реальный `DATABASE_URL` был передан в рабочем чате. Такой URL содержит логин, пароль, хост и порт БД, поэтому его нельзя вставлять в код, документацию или commit history.

### Урок

Даже если секрет нужен для настройки deploy, репозиторий должен хранить только имя переменной окружения и пример без реального значения. Реальное значение должно жить в Railway Variables или локальном `.env`, который исключен из git.

### Новое правило

Перед каждым коммитом с env-настройками запускать поиск по характерным частям секретов (`postgresql://`, реальные хосты, токены). Если секрет уже попал в публичное место, его нужно ротировать.

## 2026-05-02 — VPS для Telegram нужно проверять по сети до настройки

### Ошибка

При попытке переноса на REG.RU VPS проект был установлен и `.env` настроен, но запуск упал на `ETIMEDOUT` при запросе к `https://api.telegram.org/.../getMe`. Время ушло на серверную настройку до проверки главного риска: доступности Telegram API с выбранного хостинга.

### Урок

Для Telegram-бота на polling публичный IP не нужен, но исходящий доступ к `api.telegram.org:443` критичен. На некоторых VPS/сетях это может быть нестабильно или заблокировано.

### Новое правило

Перед переносом Telegram-бота на новый VPS первым делом проверять сеть командой `curl -I --connect-timeout 10 https://api.telegram.org`. Только после успешной проверки ставить Node.js, PM2 и деплоить приложение.
